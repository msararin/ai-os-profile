# Web Governance Discipline - sararin.ai
## Constitution for Public-Facing Changes

**Created:** 2026-05-24  
**Status:** ACTIVE  
**Applies to:** All public pages, navigation, metrics, positioning  
**Enforcement:** Every change must pass this contract before deploy

---

## Core Principles (The Constitution)

### 1. AIOS First / Lyn Credibility Second
- **Balance:** 80% AIOS / 20% Lyn
- **Primary job:** Prove AIOS governance capability, not sell Lyn biography
- **Target reader:** Senior executives evaluating AI transformation leaders
- **First impression:** "This person built a working AIOS" NOT "This person has a nice resume"

### 2. Preserve Before Improve
- **Rule:** Approved concepts must not disappear unless explicitly superseded
- **When feedback arrives:** Treat as PATCH (add/improve) not REPLACEMENT (redesign)
- **Challenge required:** If feedback conflicts with approved identity, STOP and clarify
- **Anti-fatigue:** Every change must reduce Lyn's future explanation burden

### 3. Evidence Over Claims
- **No unsupported hero metrics:** Every performance/ROI/speed/cost claim must have:
  - Baseline (compared to what?)
  - Date measured (when?)
  - Same task (apples-to-apples?)
  - Evidence source (file path or methodology)
  - Caveat (scope, limitations)
- **If evidence incomplete:** Remove, hide, or label as "prototype observation pending validation"
- **Purpose:** Site proves governance discipline, not hype creation

### 4. Page Purpose Contract
- **Every page must have:**
  - Clear job (what must reader believe after 90 seconds?)
  - Target reader (who is this for?)
  - Approved positioning (what must remain?)
- **Home page job:** Show AIOS exists and works, establish Lyn credibility just enough
- **About page job:** Prove why Lyn can build/govern AIOS (not life story)

### 5. Accurate File-Action Reporting
- **Precision required:** Distinguish "no files edited" vs "no website files edited, KB updated"
- **Why:** Building evidence/source-of-truth discipline
- **Source of truth matters:** KB changes, skill changes, config changes are real file actions

---

## Forbidden Actions (Without Explicit Override)

❌ Remove approved concepts without supersession note  
❌ Use performance metrics without evidence context  
❌ Redesign navigation without preservation check  
❌ Deploy without quality gate passing  
❌ Report "no changes" when KB/skills/configs were modified  
❌ Optimize for "looks polished" over "serves job"  
❌ Treat feedback as "replace" instead of "patch"  
❌ Add unprofessional content (e.g., `สีมงคลวันศุกร์`)  

---

## Required Actions (Before Every Change)

✅ Check must-preserve registry  
✅ Produce pre-change gate report  
✅ Declare intent before implementation  
✅ Identify files touched vs not touched  
✅ Run validation after change  
✅ Pass public-facing quality gate  
✅ Get human release approval (Lyn/Robert)  

---

## Definition of Done (Public Website Changes)

**A change is NOT done until:**

1. ✅ Pre-change gate report completed
2. ✅ Must-preserve registry checked
3. ✅ Metric evidence validated (or removed/relabeled)
4. ✅ Home first-impression checked (if affected)
5. ✅ No unsupported hero claim
6. ✅ No approved concept removed without note
7. ✅ Validation run passed
8. ✅ Changed files listed
9. ✅ Rollback path known
10. ✅ Lyn/Robert release gate passed

**Then and only then:** Commit and deploy.

---

## Escalation Criteria

**STOP and ask Lyn if:**
- Feedback conflicts with approved identity
- Approved concept must be removed to proceed
- Reputation risk detected (first impression broken)
- Evidence for hero metric is incomplete
- Unsure if change is PATCH or REPLACEMENT

**Do NOT:**
- Ask Lyn to decide every small step
- Complete small bounded tasks and report together
- Use judgment for low-risk changes within contract

---

## Anti-Patterns to Avoid

🚫 **Copywriter behavior:** Optimize for impressive output  
✅ **Architect behavior:** Optimize for agreed purpose

🚫 **Task-first:** Do many easy patches  
✅ **Risk-first:** Stabilize highest-risk issue first

🚫 **Number swap:** Change 1,750 to 1,400 without evidence  
✅ **Evidence discipline:** Remove unsupported metric or relabel

🚫 **Polish elsewhere:** Add content to About when Home fails  
✅ **First impression:** Fix where visitor lands first

🚫 **Memory reliance:** "Risa will remember"  
✅ **Contract enforcement:** "Change must pass gate to ship"

## Governance Layers (5-Layer Model)

```
1. Memory/Rule (this document)
   ↓
2. Checklist (must-preserve-registry.md)
   ↓
3. Source-of-truth file (approved state)
   ↓
4. Automated gate (scripts, future)
   ↓
5. Human release gate (Lyn/Robert approval)
```

**All 5 layers required for durable discipline.**
